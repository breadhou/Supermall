# 电商平台设计文档

## 项目概述

从单体架构起步的电商系统，核心聚焦秒杀高并发场景（万级 QPS），完成基础功能后演进至 Spring Cloud Alibaba 微服务架构。定位为简历精品项目。

**技术栈：** Java 17 + Spring Boot 3.4.4 + MyBatis-Plus 3.5.10 + MySQL 8.0 + Redis 7.x + RabbitMQ 3.12.x + Spring Security 6.x + JWT 0.13.0 + Hutool 5.8.47 + Knife4j 4.5.0

**演进路线：** 单体 → Spring Cloud Alibaba 分布式（Nacos + Sentinel + Gateway + Dubbo/Feign）

## 1. 系统架构（单体阶段）

### 1.1 分层架构

```
前端 → Nginx → Spring Boot → MySQL / Redis / RabbitMQ
```

- **前端层：** Vue/React SPA，Nginx 反向代理 + 动静分离
- **网关层：** Spring Security + JWT 认证过滤器 + 全局异常处理
- **业务层：** 按模块分包（user / product / order / cart / promotion / payment / merchant / admin）
- **基础设施层：** 封装 Redis 缓存/分布式锁、RabbitMQ 消息发送/消费
- **数据层：** MySQL 主从读写分离 + Redis 缓存

## 2. 秒杀高并发方案

### 2.1 请求处理流程

```
① 前端防抖 + 验证码
  → ② Nginx 动静分离 + IP 限流
    → ③ 网关层令牌桶限流
      → ④ Redis Lua 原子预减库存
        → ⑤ 库存不足 → 直接返回失败
        → ⑥ 库存足够 → 发送 MQ 消息
          → ⑦ 消费者异步下单（MySQL 扣库存 + 生成订单）
            → ⑧ 前端轮询获取下单结果
```

### 2.2 各层策略

**前端层：**
- 秒杀按钮点击后置灰 + 倒计时，防止重复提交
- 数学验证码/滑块验证码
- 商品详情页静态化，CDN 缓存

**网关层：**
- IP 级别黑名单（同 IP 每秒超阈值自动拉黑）
- 令牌桶限流（Guava RateLimiter，后续切换到 Sentinel）
- JWT 校验

**服务层（核心）：**
- 活动开始前，秒杀库存预热到 Redis
- 下单时，Lua 脚本原子执行 `decr`，判断是否超卖
- Redis 扣减成功 ≠ 下单成功，进入 MQ 异步处理队列

**MQ 异步下单：**
- 秒杀成功消息发送到 RabbitMQ（带唯一消息 ID 防重）
- 消费者逐条：校验库存 → 扣 MySQL → 生成订单 → 更新缓存状态
- 消费失败 → 死信队列 + 补偿回滚 Redis 库存

**结果通知：**
- 前端轮询订单状态接口（后续可升级 WebSocket）
- Redis 记录秒杀结果缓存，避免重复查库

### 2.3 超卖防护（三层）

1. Redis Lua 脚本原子操作
2. MySQL 乐观锁：`UPDATE ... WHERE stock >= ?`
3. seckill_order 唯一约束：`uk_user_seckill (user_id, seckill_item_id)`

### 2.4 秒杀与优惠券互斥

秒杀商品不能使用优惠券。下单逻辑中秒杀订单跳过优惠券校验流程。

## 4. API 设计

### 4.1 统一响应格式

```json
{
  "code": 200,
  "message": "success",
  "data": { ... },
  "timestamp": 1706000101000
}
```

### 4.2 核心端点

**认证：**
- `POST /api/auth/register` · `POST /api/auth/login` · `GET /api/auth/refresh`

**商品：**
- `GET /api/products` · `GET /api/products/{id}` · `GET /api/products/{id}/skus` · `GET /api/categories` · `GET /api/products/{id}/reviews`

**购物车 & 订单：**
- `GET /api/cart` · `POST /api/cart/items` · `PUT /api/cart/items/{id}` · `DELETE /api/cart/items/{id}`
- `POST /api/orders` · `GET /api/orders` · `GET /api/orders/{orderNo}`

**秒杀：**
- `GET /api/seckill/activities` · `GET /api/seckill/items/{activityId}`
- `GET /api/seckill/{itemId}/countdown` — 返回剩余库存和活动状态
- `POST /api/seckill/{itemId}/order` — 核心秒杀下单（无参，JWT 解析用户）
- `GET /api/seckill/result/{itemId}` — 轮询秒杀结果

**优惠券：**
- `GET /api/coupons` · `POST /api/coupons/{id}/claim` · `GET /api/coupons/my`

### 4.3 接口规范

- Controller 只做参数校验和路由，不写业务逻辑
- Service 接口 + 单一实现类，方便后续 RPC 切换
- 秒杀接口走独立限流，不与普通下单共用
- 接口文档通过 Knife4j 自动生成

## 5. 编码规范与约定

### 5.1 命名规范

- Redis Key：`mall:seckill:stock:{itemId}`、`mall:seckill:result:{itemId}:{userId}`
- MQ 队列：`mall.seckill.order`（队列）、`mall.seckill.order.dlx`（死信）
- 统一异常：`BusinessException`，GlobalExceptionHandler 全局兜底
- 数据库操作：MyBatis-Plus BaseMapper，复杂 SQL 写 XML

### 5.2 分层约束

| 层 | 所在模块 | 职责 | 禁止 |
|----|---------|------|------|
| Controller | mall-server | 参数校验、路由、响应封装 | 不写业务逻辑 |
| Service | mall-server | 业务编排、事务管理 | 不直接操作 Redis/MQ |
| Mapper | mall-server | 数据库访问 | 不写业务逻辑 |
| Infrastructure | mall-infra | Redis/MQ 操作封装 | 不包含业务判断 |
| Security | mall-security | JWT 生成/校验、认证过滤 | 不包含业务规则 |
| Common | mall-common | 响应体、异常、工具类 | 不依赖其他业务模块 |

### 5.3 对象转换

- PO（Persistent Object）：对应数据库表，Mapper 层使用
- VO（View Object）：返回前端，Controller 层使用
- DTO（Data Transfer Object）：Service 间传输，查询参数封装

## 6. 演进路线

### 阶段一：单体（当前目标）

- 单应用部署 `mall-server.jar`
- Redis 单实例缓存，MySQL 读写分离
- RabbitMQ 异步解耦秒杀链路
- 完成所有业务功能 + 秒杀压测验证
- 单元测试覆盖率 > 60%

### 阶段二：分布式化

| 组件 | 用途 |
|------|------|
| Nacos | 注册中心 + 配置中心 |
| Gateway | 统一网关 |
| Sentinel | 限流熔断 |
| Dubbo/Feign | 服务间 RPC |
| ShardingSphere | 订单表分库分表 |
| Seata | 分布式事务 |

**微服务拆分优先级（按模块）：**

1. mall-seckill（秒杀—高频高并发，最先拆）
2. mall-order（订单+购物车+退款）
3. mall-product（商品+分类+库存+评价）
4. mall-user（用户+认证+地址）
5. mall-promotion（优惠券）
6. mall-gateway（网关独立服务）
7. mall-merchant / mall-admin（后台管理—最后拆）

每个微服务独立数据库，共享 mall-common 和 mall-infra 包。

## 7. 非功能需求

| 类别 | 目标 |
|------|------|
| 秒杀 QPS | 单机 5000+，集群 10000+ |
| 接口响应 | 普通接口 P99 < 200ms，秒杀接口 P99 < 500ms |
| 数据一致性 | 秒杀不超卖，订单不丢消息（MQ 确认 + 死信补偿） |
| 幂等性 | 下单/支付接口天然幂等 |
| 安全性 | JWT 过期刷新、XSS 过滤、SQL 参数化 |
